What is Dcanary?

Dcanary is a decentralized CI/CD pipeline system built on the Internet Computer Protocol (ICP). It provides trustless, tamper-proof build verification through distributed consensus and cryptographic verification. Named after Daniel (its creator) and the "canary in the coal mine" concept for early issue detection.

Security First

Every build is cryptographically verified across multiple independent executors, ensuring no single point of failure or tampering.

Decentralized

Built on ICP, Dcanary operates without centralized control, providing true decentralization and censorship resistance.

Developer Friendly

Simple CLI interface and GitHub integration make it easy to integrate into existing workflows.

Quick Start

5-Minute Setup
$ git clone https://github.com/username/dcanary.git
$ cd dcanary/Dcanary && npm install
$ dfx start --background && dfx deploy
$ cd cli && npm install -g .

git clone https://github.com/dcanary/dcanary.git

Execution Pipeline:

  1. Receive build request from verification canister
  2. Fetch build instructions from instructions canister
  3. Set up isolated execution environment
  4. Execute build steps with timeout protection
  5. Capture artifacts and generate cryptographic hash
  6. Return results to verification canister

Security Model

Zero Trust Architecture

Dcanary operates on a zero-trust security model where no single entity is trusted. All build results are independently verified across multiple executors.

Cryptographic Verification

SHA-256 Hashing

All build artifacts are hashed using SHA-256

Digital Signatures

Executors sign their results using IC identity

Timestamp Verification

All operations are timestamped on-chain

Consensus Mechanism

Multi-Executor Agreement

Minimum 3 executors must agree on build results

Weighted Voting

Executors have reputation-based voting weights

Dispute Resolution

Automatic handling of conflicting results

Consensus Mechanism

How Dcanary ensures build integrity through distributed consensus algorithms.

Byzantine Fault Tolerance

Dcanary uses a Byzantine Fault Tolerant consensus mechanism to handle up to 33% malicious or faulty executors while maintaining system integrity and reliability.

Executor Pool

Minimum Executors: 3
Recommended: 5-7
Max Faults: ⌊(n-1)/3⌋

Voting Weights

New Executor: 1.0
Verified Executor: 1.5
Trusted Executor: 2.0

Consensus Rules

Agreement Threshold: 67%
Timeout: 10 min
Retry Limit: 3

Consensus Algorithm

1

Proposal Phase

Build instructions are distributed to all available executors

2

Execution Phase

Each executor independently builds and generates cryptographic hashes

3

Security Best Practices

Build Script Security

Avoid Sensitive Data

Never include API keys, passwords, or sensitive configuration in build scripts. Use environment variables or secure configuration management.

Input Validation

Validate all inputs and parameters to prevent injection attacks. Use parameterized commands and avoid dynamic command construction.

Network Access

Minimize network access during builds. Use verified package sources and consider offline builds for sensitive projects.

Access Control

Principal Management

Use dedicated principals for CI/CD operations. Rotate keys regularly and limit permissions to minimum required.

Project Permissions

Carefully manage project collaborators. Use role-based access control and regular permission audits.

Verification Guidelines

Executor Diversity

Use geographically and organizationally diverse executors. Minimum 3 executors recommended for production.

Build Reproducibility

Ensure builds are deterministic. Pin all dependencies, avoid timestamps, and use consistent environments.

2. Build Script (scripts/build.sh)


#!/bin/bash
set -e

echo "🚀 TypeScript Build Process Started"

# Environment setup
export NODE_ENV=production
export BUILD_TIMESTAMP=$(date -u +%Y%m%d%H%M%S)

# Install dependencies with exact versions
echo "📦 Installing dependencies..."
npm ci --only=production

# Type checking
echo "🔍 Type checking..."
npx tsc --noEmit

# Linting
echo "🎯 Linting..."
npx eslint src --ext .ts,.tsx --fix

# Testing
echo "🧪 Running tests..."
npm test -- --coverage --ci

# Build
echo "🏗️  Building..."
npm run build

# Generate build manifest
echo "📝 Generating build manifest..."
BUILD_HASH=$(find dist -type f -exec sha256sum {} \; | sort | sha256sum | cut -d' ' -f1)
cat > build-manifest.json << EOF
{
  "project": "my-typescript-app",
  "version": "${BUILD_VERSION:-1.0.0}",
  "timestamp": "${BUILD_TIMESTAMP}",
  "build_hash": "${BUILD_HASH}",
  "node_version": "$(node --version)",
  "npm_version": "$(npm --version)"
}
EOF

# Package artifacts
echo "📦 Packaging artifacts..."
tar -czf dist.tar.gz dist/ build-manifest.json
sha256sum dist.tar.gz > dist.tar.gz.sha256

echo "✅ Build completed successfully!"
echo "📊 Build hash: ${BUILD_HASH}"

Common Build Issues

Non-Deterministic Builds

Problem: Different executors produce different build hashes

Solutions:

  • • Remove timestamps from build outputs
  • • Pin all dependencies to exact versions
  • • Use consistent file ordering in archives
  • • Set fixed build environment variables

Dependency Resolution Failures

Problem: Builds fail due to missing or incompatible dependencies

Solutions:

  • • Use lockfiles (package-lock.json, yarn.lock)
  • • Specify exact version numbers
  • • Include system dependencies in requirements
  • • Test builds in clean environments

Network Timeouts

Problem: Builds fail due to network connectivity issues

Solutions:

  • • Increase timeout values
  • • Use reliable package mirrors
  • • Implement retry logic in build scripts
  • • Consider offline/cached builds

class CustomRustAgent implements BuildAgent {
  async execute(instructions: BuildInstructions): Promise {
    // Custom Rust build logic
    const result = await this.runRustBuild(instructions);
    return {
      success: result.exitCode === 0,
      artifacts: result.artifacts,
      buildHash: this.computeHash(result.artifacts),
      logs: result.logs
    };

    return {
      supportedLanguages: ['rust'],
      maxConcurrentBuilds: 5,
      specialFeatures: ['cargo-cache', 'cross-compile']
    };
  }

├── services/

version: "1.0"

services:
  - name: "api"
    path: "services/api"
    triggers:
      - "services/api/**"
      - "shared/**"
  - name: "web"
    path: "services/web"
    triggers:
      - "services/web/**"
      - "shared/**"
  - name: "worker"
    path: "services/worker"
    triggers:
      - "services/worker/**"
      - "shared/**"

build:
  strategy: "affected-only"
  parallel: true
  dependencies:
    - "shared"

Frequently Asked Questions

How does Dcanary ensure build integrity?

Dcanary uses multiple independent executors to build your code. Each executor generates a cryptographic hash of the build output. The system requires consensus among executors before marking a build as verified.

What happens if an executor fails or produces different results?

If an executor fails or produces different results, the verification canister will automatically select additional executors to maintain the minimum required for consensus. Consistently failing executors are flagged and may be removed from the pool.

Can I use Dcanary with private repositories?

Yes, Dcanary supports private repositories. You can configure access tokens for GitHub, GitLab, or other SCM providers. The build instructions and source code are encrypted during transmission and execution.

How much does it cost to run Dcanary?

Dcanary runs on the Internet Computer, so costs are based on cycles consumption. Typical costs range from $0.01-$0.10 per build depending on complexity and number of executors. You only pay for actual usage.

Can I run my own executors?

Yes, you can deploy and register your own executors. This gives you full control over the build environment while still benefiting from the decentralized verification system.
